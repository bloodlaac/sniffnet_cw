# Техническое задание
## Часть 1. Общие положения и границы проекта

### 1. Наименование системы
Веб-приложение **SniffNet** для оценки качества продуктов питания по загруженному изображению.

### 2. Основание для разработки
Основанием для разработки является учебная курсовая работа по дисциплине веб-разработки на Java Spring.

### 3. Предметная область
Система предназначена для определения степени свежести продуктов питания по фотографии, загружаемой пользователем. Система должна обеспечивать хранение пользователей, экспериментов обучения, обученных моделей, метрик моделей, результатов классификации и связанных служебных данных.

### 4. Цель разработки
Целью разработки является создание клиент-серверной информационной системы, которая:
- обеспечивает регистрацию и аутентификацию пользователей;
- предоставляет доступ к созданию и просмотру экспериментов обучения нейронной сети;
- позволяет пользователю выбрать обученную модель из списка;
- отображает метрики обученной модели;
- позволяет загрузить фотографию продукта и получить прогноз свежести;
- предоставляет администратору средства управления учетными записями пользователей.

### 5. Объект автоматизации
Автоматизируется процесс:
- ведения учетных записей пользователей;
- управления данными об экспериментах обучения;
- выбора и отображения обученных моделей;
- получения результата классификации изображения продукта из внешнего Python-микросервиса;
- хранения истории обращений к системе и результатов работы.

### 6. Состав системы
Система состоит из следующих подсистем:
1. **Frontend-подсистема** на Angular.
2. **Backend-подсистема** на Java 17 и Spring Boot 4.x.
3. **Подсистема хранения данных** на PostgreSQL.
4. **Подсистема интеграции с Python-модулем обучения**.
5. **Подсистема интеграции с Python-микросервисом классификации**.
6. **Подсистема документирования API** на Swagger / OpenAPI.
7. **Подсистема контейнеризации и локального развертывания** на Docker Compose.

### 7. Технологические ограничения
Разработка должна соответствовать следующим обязательным ограничениям:
- язык серверной разработки: **Java 17**;
- серверный фреймворк: **Spring Boot 4.x**;
- ORM и доступ к данным: **Hibernate, JPA**;
- база данных: **PostgreSQL**;
- клиентская часть: **Angular**;
- взаимодействие клиента и сервера: **REST API**;
- должна быть реализована **авторизация**;
- должны быть реализованы **CRUD-операции** для предусмотренных сущностей;
- должны быть реализованы **валидация** и **обработка ошибок**;
- загрузка файлов должна выполняться через **REST API**.

### 8. Дополнительные требования
В проект рекомендуется включить:
- Swagger / OpenAPI для описания REST API;
- Docker Compose для локального запуска окружения.

### 9. Заинтересованные стороны
- **Пользователь** — выполняет регистрацию, вход, просмотр доступных моделей, создание экспериментов, загрузку фотографий, запуск классификации, просмотр результатов.
- **Администратор** — выполняет вход, просмотр и управление учетными записями пользователей.
- **Python-модуль обучения** — внешний по отношению к курсовой компонент, запускающий обучение модели.
- **Python-микросервис классификации** — внешний по отношению к курсовой компонент, возвращающий результат распознавания и вероятности по классам.

### 10. Роли пользователей
В системе должны быть реализованы две роли:
- **ROLE_USER** — базовый пользователь;
- **ROLE_ADMIN** — администратор.

### 11. Границы проекта
В рамки курсовой работы входят:
- проектирование и разработка Java Spring backend;
- проектирование и разработка Angular frontend;
- проектирование базы данных PostgreSQL;
- проектирование REST API;
- разработка JWT-аутентификации и авторизации;
- загрузка изображений через REST API;
- интеграция с Python-компонентами через HTTP-контракты либо через mock-реализации;
- отображение метрик моделей и результатов классификации.

В рамки курсовой работы **не входят**:
- реализация алгоритмов машинного обучения;
- реализация Python-модуля обучения;
- реализация Python-микросервиса классификации;
- обучение нейросети внутри Java-приложения;
- хранение фактических файлов `model.pth` в PostgreSQL;
- промышленное отказоустойчивое развертывание.

### 12. Принятые проектные допущения
Для устранения неоднозначностей принимаются следующие допущения:
1. Java-приложение является основным backend-компонентом системы.
2. Python-компоненты считаются внешними сервисами с фиксированными интеграционными контрактами.
3. На ранних этапах разработки вместо реального Python-сервиса допускается mock-реализация, возвращающая предопределенные данные.
4. JWT применяется как основной механизм аутентификации API.
5. Доступ к функциональности осуществляется только после входа в систему, кроме экранов регистрации и аутентификации.
6. Выбор модели для классификации выполняется из перечня доступных обученных моделей.
7. Метрики модели, отображаемые пользователю, поступают из Python-сервиса либо из синхронизированной базы данных Java backend.
8. Файлы весов `model.pth` хранятся локально на машине пользователя или в файловой структуре Python-модуля; Java backend хранит только метаданные модели и ссылки либо идентификаторы.
9. Физическая модель данных из Python-модуля принимается как базовая и расширяется минимально необходимыми сущностями для покрытия требований веб-приложения.

### 13. Архитектурный стиль
Для системы принимается трехуровневая архитектура:
- уровень представления — Angular frontend;
- уровень бизнес-логики — Spring Boot backend;
- уровень данных — PostgreSQL.

Внешние Python-сервисы подключаются к уровню бизнес-логики по REST.

### 14. Основные пользовательские сценарии
1. Пользователь открывает приложение.
2. Пользователь выполняет регистрацию или вход.
3. Пользователь получает JWT-токен и доступ к защищенным разделам.
4. Пользователь просматривает список экспериментов.
5. Пользователь создает новый эксперимент обучения.
6. Backend передает запрос на запуск обучения во внешний Python-компонент либо в mock-слой.
7. После завершения обучения пользователь просматривает метрики и связанную модель.
8. Пользователь открывает экран проверки по фото.
9. Пользователь выбирает модель, загружает изображение и запускает прогноз.
10. Система получает результат из Python-сервиса и отображает итоговый класс и вероятности.
11. Администратор просматривает и изменяет данные пользователей.

### 15. Термины и определения
- **Эксперимент** — зафиксированный запуск обучения модели на определенном датасете и наборе гиперпараметров.
- **Конфигурация обучения** — набор параметров обучения: количество эпох, batch size, learning rate, optimizer, loss function и дополнительные параметры.
- **Модель** — обученная нейросетевая модель, доступная для использования при классификации.
- **Метрики модели** — числовые показатели качества обучения, например accuracy и loss.
- **Классификация** — получение результата свежести продукта по изображению.
- **JWT** — JSON Web Token, применяемый для подтверждения подлинности запросов.

### 16. Ожидаемый результат разработки
Результатом разработки должен быть программный комплекс, который локально запускается, подключается к PostgreSQL, предоставляет Angular-интерфейс и REST API, поддерживает регистрацию, вход, управление пользователями, создание экспериментов, просмотр моделей и метрик, загрузку изображений и получение классификации.

# Техническое задание
## Часть 2. Функциональные требования

### 1. Общие функциональные требования
Система должна обеспечивать:
- регистрацию пользователя;
- аутентификацию пользователя;
- авторизацию по ролям;
- управление учетными записями пользователей;
- создание и просмотр экспериментов обучения;
- получение и отображение статуса эксперимента;
- получение и отображение метрик обучения;
- выбор обученной модели;
- загрузку изображения продукта;
- получение результата классификации по изображению;
- сохранение истории классификаций;
- CRUD-операции для сущностей, входящих в предметную область.

### 2. Требования к модулю регистрации и аутентификации
#### 2.1. Регистрация
Система должна предоставлять возможность регистрации нового пользователя.

При регистрации пользователь обязан указать:
- имя пользователя;
- адрес электронной почты;
- пароль.

Требования:
- имя пользователя должно быть уникальным;
- электронная почта должна быть уникальной;
- пароль должен храниться только в хэшированном виде;
- при успешной регистрации должна создаваться учетная запись с ролью `ROLE_USER`;
- после регистрации пользователь должен иметь возможность выполнить вход в систему.

#### 2.2. Аутентификация
Система должна предоставлять возможность входа по логину и паролю.

При успешной аутентификации backend должен:
- проверить учетные данные;
- сформировать JWT-токен;
- вернуть токен клиентскому приложению;
- вернуть базовую информацию о пользователе и его роли.

#### 2.3. Выход из системы
Система должна поддерживать клиентский выход из системы путем удаления JWT-токена на стороне клиента. При необходимости может быть реализован серверный blacklist токенов, но это не является обязательным минимумом курсовой работы.

### 3. Требования к авторизации
Система должна разграничивать доступ по ролям.

#### 3.1. Права пользователя
Пользователь с ролью `ROLE_USER` должен иметь право:
- просматривать собственный профиль;
- просматривать список доступных датасетов;
- создавать новый эксперимент;
- просматривать список экспериментов;
- просматривать детали эксперимента;
- просматривать список доступных моделей;
- просматривать метрики выбранной модели;
- загружать изображение продукта;
- инициировать классификацию изображения;
- просматривать результат классификации;
- просматривать собственную историю классификаций.

#### 3.2. Права администратора
Пользователь с ролью `ROLE_ADMIN` должен иметь право:
- выполнять все действия `ROLE_USER`, если это не ограничено отдельно;
- просматривать список пользователей;
- искать пользователей;
- просматривать карточку пользователя;
- изменять данные пользователя;
- блокировать или деактивировать пользователя, если это предусмотрено реализацией;
- удалять пользователя;
- изменять роль пользователя.

### 4. Требования к модулю пользователей
Система должна предоставлять CRUD-операции для учетных записей пользователей.

#### 4.1. Просмотр пользователей
Администратор должен видеть список пользователей с основными полями:
- идентификатор;
- username;
- email;
- дата создания;
- роль;
- статус активности.

#### 4.2. Редактирование пользователя
Администратор должен иметь возможность:
- изменить username;
- изменить email;
- изменить роль;
- изменить статус активности.

#### 4.3. Удаление пользователя
Администратор должен иметь возможность удалить пользователя.

При удалении должны быть определены правила обработки связанных данных:
- либо физическое удаление пользователя при отсутствии ограничений целостности;
- либо предварительное удаление зависимых записей;
- либо запрет удаления при наличии критически связанных сущностей.

Для курсовой допускается физическое удаление пользователя, если схема и бизнес-правила это разрешают.

### 5. Требования к модулю датасетов
Система должна предоставлять список датасетов, доступных для запуска обучения.

Для каждого датасета должны храниться и отображаться:
- идентификатор датасета;
- название;
- количество классов;
- источник.

Пользователь должен иметь возможность выбрать датасет при создании нового эксперимента.

### 6. Требования к модулю конфигурации обучения
При создании нового эксперимента пользователь должен иметь возможность задать конфигурацию обучения.

Обязательные поля конфигурации:
- `epochs`;
- `batchSize`;
- `learningRate`;
- `optimizer`;
- `lossFunction`;
- `validationSplit`.

Дополнительные поля, поддерживаемые на уровне схемы и интеграции:
- `layersNum`;
- `neuronsNum`.

Требования:
- пользователь должен иметь возможность ввести значения вручную;
- система должна проверять корректность диапазонов значений;
- конфигурация должна сохраняться в базе данных;
- конфигурация должна быть связана с экспериментом.

### 7. Требования к модулю экспериментов
#### 7.1. Создание эксперимента
Пользователь должен иметь возможность создать новый эксперимент обучения.

При создании эксперимента система должна:
1. принять выбранный датасет;
2. принять параметры конфигурации;
3. создать запись конфигурации обучения;
4. создать запись эксперимента;
5. установить начальный статус эксперимента;
6. передать запрос во внешний Python-компонент либо в mock-реализацию.

#### 7.2. Список экспериментов
Система должна отображать список экспериментов.

Для каждого эксперимента должны отображаться:
- идентификатор;
- название датасета;
- дата и время начала;
- дата и время завершения;
- статус.

#### 7.3. Детали эксперимента
Для выбранного эксперимента система должна отображать:
- общую информацию;
- используемый датасет;
- пользователя, создавшего эксперимент;
- период выполнения;
- статус;
- конфигурацию обучения;
- связанную модель;
- итоговые метрики;
- отчет эксперимента.

#### 7.4. Статусы эксперимента
Эксперимент должен поддерживать как минимум следующие статусы:
- `CREATED` — создан;
- `RUNNING` — в процессе обучения;
- `COMPLETED` — завершен успешно;
- `FAILED` — завершен с ошибкой.

#### 7.5. Отчет эксперимента
Если Python-компонент возвращает графики или отчет по эпохам, система должна уметь сохранять и отображать отчет эксперимента. В базовой версии допускается хранение URL, файла изображения или JSON с данными графика.

### 8. Требования к модулю моделей
Система должна предоставлять список доступных обученных моделей.

Для каждой модели должны отображаться:
- идентификатор модели;
- название модели;
- датасет;
- конфигурация обучения;
- ключевые метрики;
- длительность обучения;
- признак доступности для инференса.

Правила:
- модель может использоваться для классификации только после успешного завершения эксперимента;
- модель должна быть привязана к эксперименту, датасету и конфигурации;
- источник весов модели находится вне Java backend.

### 9. Требования к модулю классификации по фото
#### 9.1. Выбор модели
Перед загрузкой изображения пользователь должен выбрать модель из списка.

При выборе модели система должна показывать:
- наименование модели;
- accuracy;
- loss;
- дополнительные метрики, если они доступны.

#### 9.2. Загрузка изображения
Пользователь должен иметь возможность загрузить изображение продукта.

Требования:
- поддерживаемые форматы: как минимум `JPG`, `JPEG`, `PNG`;
- изображение передается через REST API;
- изображение должно сохраняться в системе;
- должно существовать ограничение на максимальный размер файла;
- при попытке загрузить файл недопустимого формата система должна вернуть ошибку валидации.

#### 9.3. Получение прогноза
После выбора модели и загрузки изображения пользователь должен иметь возможность отправить запрос на классификацию.

Система должна:
1. принять файл и идентификатор модели;
2. передать данные во внешний Python-сервис классификации либо в mock-слой;
3. получить ответ с результатом;
4. сохранить результат в истории классификаций;
5. показать пользователю итоговый класс и вероятности по классам.

#### 9.4. Отображение результата
Минимально отображаемые данные:
- итоговый класс, например `Свежий` или `Испорченный`;
- общая уверенность прогноза;
- распределение вероятностей по классам;
- выбранная модель;
- ссылка на загруженное изображение или миниатюра изображения.

### 10. Требования к модулю истории классификаций
Система должна хранить историю запросов на классификацию.

Для каждой записи должны сохраняться:
- идентификатор пользователя;
- идентификатор модели;
- путь к изображению или идентификатор файла;
- дата и время запроса;
- итоговый класс;
- уверенность;
- вероятности по классам;
- служебный статус выполнения запроса.

### 11. Требования к поиску и фильтрации
Система должна поддерживать:
- поиск пользователя по username для администратора;
- фильтрацию экспериментов по статусу;
- фильтрацию экспериментов по пользователю;
- фильтрацию моделей по датасету;
- фильтрацию истории классификаций по пользователю и дате.

### 12. Требования к навигации интерфейса
Система должна содержать как минимум следующие разделы интерфейса:
- вход;
- регистрация;
- главная страница;
- список экспериментов;
- страница создания нового эксперимента;
- страница деталей эксперимента;
- страница проверки по фото;
- административный раздел пользователей.

### 13. Требования к mock-режиму
В mock-режиме система должна позволять:
- создавать эксперимент без фактического запуска обучения;
- переводить эксперимент в статус `COMPLETED` или `FAILED` по сценариям заглушки;
- возвращать заранее подготовленные метрики;
- возвращать заранее подготовленный результат классификации.

### 14. Требования к журналированию действий
Система должна журналировать как минимум:
- попытки входа;
- создание эксперимента;
- обращения к внешним Python-сервисам;
- ошибки интеграции;
- ошибки валидации;
- ошибки безопасности.

# Техническое задание
## Часть 3. Интеграция с Python-компонентами

### 1. Общие положения
Python-компоненты не входят в состав курсовой работы, однако Java backend обязан быть спроектирован так, чтобы:
- корректно интегрироваться с модулем обучения;
- корректно интегрироваться с микросервисом классификации;
- позволять замену реальных Python-компонентов на mock-реализации без изменения публичного REST API Java backend.

### 2. Внешние компоненты интеграции
#### 2.1. Python-модуль обучения
Внешний компонент, отвечающий за запуск обучения модели по выбранному датасету и конфигурации обучения.

Ожидаемый результат его работы:
- создание или обновление данных об эксперименте;
- создание обученной модели;
- формирование метрик качества;
- при наличии — формирование отчета по эпохам;
- сохранение файлов весов `model.pth` вне Java backend.

#### 2.2. Python-микросервис классификации
Внешний компонент, отвечающий за классификацию изображения продукта на основе выбранной модели.

Ожидаемый результат его работы:
- итоговый класс;
- вероятность итогового класса;
- вероятности по всем классам;
- служебная информация о модели и выполнении запроса.

### 3. Общие требования к интеграции
1. Java backend должен использовать отдельный сервисный слой интеграции.
2. Контракты внешних вызовов должны быть инкапсулированы в adapter-классах.
3. Все интеграционные ошибки должны перехватываться и преобразовываться в контролируемые ответы API.
4. Должна быть предусмотрена возможность таймаута внешнего вызова.
5. Должна быть предусмотрена возможность mock-режима по конфигурации приложения.
6. Java backend не должен содержать кода обучения нейросети.

### 4. Контракт интеграции с сервисом обучения
#### 4.1. Входные данные на запуск обучения
Java backend должен передавать как минимум:
- `experimentId`;
- `datasetId`;
- `datasetName`;
- `classesNum`;
- `configId`;
- `epochsNum`;
- `batchSize`;
- `learningRate`;
- `optimizer`;
- `lossFunction`;
- `validationSplit`;
- `layersNum` и `neuronsNum`, если используются;
- `userId` инициатора запуска.

#### 4.2. Ожидаемый ответ от сервиса обучения
Сервис обучения должен вернуть данные в одном из двух вариантов:

**Синхронный вариант**:
- статус выполнения;
- дата начала;
- дата завершения;
- идентификатор модели;
- название модели;
- accuracy;
- loss;
- дополнительные метрики;
- длительность обучения;
- путь или идентификатор файла отчета;
- путь или идентификатор модели в Python-среде.

**Асинхронный вариант**:
- подтверждение принятия задачи;
- внешний идентификатор задачи;
- начальный статус;
- далее обновление статуса выполняется отдельным запросом либо callback-механизмом.

Для курсовой работы допускается упрощенный синхронный либо псевдоасинхронный сценарий.

### 5. Контракт интеграции с сервисом классификации
#### 5.1. Входные данные на классификацию
Java backend должен передавать как минимум:
- `modelId`;
- `modelName`;
- ссылку, путь или бинарное содержимое изображения;
- при необходимости — метаинформацию о пользователе и формате файла.

#### 5.2. Ожидаемый ответ от сервиса классификации
Сервис классификации должен вернуть:
- `predictedClass` — итоговый класс;
- `confidence` — итоговую уверенность;
- `classProbabilities` — вероятности по классам;
- `requestId` — идентификатор запроса, если предусмотрен;
- `serviceStatus` — статус выполнения;
- `message` — диагностическое сообщение при необходимости.

### 6. Синхронизация данных между Java backend и Python-средой
Java backend должен обеспечивать согласованность следующих данных:
- сведения о датасетах;
- сведения о конфигурациях обучения;
- сведения об экспериментах;
- сведения о моделях;
- сведения о метриках.

Допускаются два варианта:
1. **Java backend как основное хранилище** — Python-компоненты только возвращают результат, а backend сохраняет его в PostgreSQL.
2. **Python-компонент как источник части данных** — backend периодически или по событию синхронизирует данные.

Для курсовой рекомендуется первый вариант.

### 7. Работа с файлами `model.pth`
1. Фактические веса моделей `model.pth` хранятся вне Java backend.
2. PostgreSQL не используется как основное хранилище файлов весов.
3. Java backend должен хранить только метаданные, необходимые для выбора модели и отображения ее характеристик.
4. Идентификатор модели в Java backend должен однозначно соответствовать модели, доступной Python-компоненту.

### 8. Работа с графиками и отчетами обучения
Если Python-компонент возвращает графики обучения, Java backend должен поддерживать один из форматов хранения:
- URL внешнего ресурса;
- путь к локальному файлу изображения;
- JSON-массив точек для последующего построения графиков во frontend;
- бинарный файл, сохраняемый в файловой системе backend.

### 9. Требования к отказоустойчивости интеграции
При сбое внешнего сервиса backend должен:
- завершить запрос контролируемой ошибкой;
- вернуть пользователю понятное сообщение;
- не нарушать целостность базы данных;
- пометить эксперимент или запрос классификации как завершенный с ошибкой, если запись уже создана.

### 10. Требования к mock-адаптеру обучения
Mock-адаптер обучения должен:
- принимать те же входные данные, что и реальный адаптер;
- создавать модель с фиктивным идентификатором;
- возвращать заранее подготовленные метрики;
- менять статус эксперимента на `COMPLETED` или `FAILED`;
- при необходимости возвращать ссылку на подготовленный график.

### 11. Требования к mock-адаптеру классификации
Mock-адаптер классификации должен:
- принимать модель и файл изображения;
- возвращать детерминированный либо псевдослучайный результат;
- возвращать итоговый класс и набор вероятностей;
- позволять демонстрировать экран результата без реального Python-сервиса.

### 12. Конфигурирование режима интеграции
В `application.yml` или `application.properties` должны быть предусмотрены параметры:
- включение mock-режима;
- URL сервиса обучения;
- URL сервиса классификации;
- таймауты соединения и чтения;
- флаг логирования интеграционных запросов.

### 13. Ограничения безопасности интеграции
При взаимодействии с Python-сервисами запрещается:
- передавать пароль пользователя;
- передавать JWT без необходимости;
- допускать исполнение произвольных путей к файлам без проверки;
- доверять ответу внешнего сервиса без базовой валидации структуры.

# Техническое задание
## Часть 4. Требования к данным и базе данных

### 1. Общие требования
В качестве СУБД должна использоваться PostgreSQL.

База данных должна обеспечивать хранение:
- пользователей и ролей;
- датасетов;
- конфигураций обучения;
- экспериментов;
- моделей;
- метрик;
- истории классификаций;
- метаданных загруженных изображений.

### 2. Базовая физическая модель
Базовая физическая модель принимается по предоставленному Python-файлу `db_models.py`, в котором определены сущности:
- `User`;
- `Experiment`;
- `Dataset`;
- `TrainingConfig`;
- `Model`;
- `Metric`.

Эта модель принимается как исходная и должна быть адаптирована для реализации веб-приложения.

### 3. Обязательные таблицы базовой модели
#### 3.1. Таблица `user`
Назначение: хранение учетных записей.

Поля:
- `user_id` — первичный ключ;
- `username` — уникальное имя пользователя;
- `email` — уникальный email;
- `password` — хэш пароля;
- `created_at` — дата создания.

#### 3.2. Таблица `dataset`
Назначение: хранение информации о датасетах.

Поля:
- `dataset_id` — первичный ключ;
- `name` — название датасета;
- `classes_num` — количество классов;
- `source` — источник датасета.

#### 3.3. Таблица `training_config`
Назначение: хранение параметров обучения.

Поля:
- `config_id` — первичный ключ;
- `epochs_num`;
- `batch_size`;
- `loss_function`;
- `learning_rate`;
- `optimizer`;
- `layers_num`;
- `neurons_num`.

#### 3.4. Таблица `experiment`
Назначение: хранение фактов запуска обучения.

Поля:
- `experiment_id` — первичный ключ;
- `dataset_id` — внешний ключ на `dataset`;
- `config_id` — внешний ключ на `training_config`;
- `user_id` — внешний ключ на `user`;
- `start_time` — дата и время начала;
- `end_time` — дата и время завершения.

#### 3.5. Таблица `model`
Назначение: хранение сведений об обученных моделях.

Поля:
- `model_id` — первичный ключ;
- `dataset_id` — внешний ключ на `dataset`;
- `config_id` — внешний ключ на `training_config`;
- `params_num` — количество параметров;
- `weights` — бинарное поле, предусмотренное исходной схемой;
- `name` — имя модели;
- `training_time` — длительность обучения.

#### 3.6. Таблица `metric`
Назначение: хранение метрик обучения.

Поля:
- `metric_id` — первичный ключ;
- `dataset_id` — внешний ключ на `dataset`;
- `config_id` — внешний ключ на `training_config`;
- `train_accuracy`;
- `train_loss`.

### 4. Необходимые расширения базовой схемы
Поскольку исходная схема не покрывает все требования веб-приложения, она должна быть расширена.

#### 4.1. Таблица `role`
Назначение: хранение ролей системы.

Поля:
- `role_id` — первичный ключ;
- `code` — код роли, например `ROLE_USER`, `ROLE_ADMIN`;
- `name` — отображаемое название роли.

#### 4.2. Таблица `user_role`
Назначение: связь пользователей и ролей.

Поля:
- `user_id` — внешний ключ на `user`;
- `role_id` — внешний ключ на `role`.

Допускается упрощение до хранения одной роли в таблице `user`, если преподаватель допускает такую денормализацию. Но предпочтительной является отдельная role-модель.

#### 4.3. Поле статуса эксперимента
В таблицу `experiment` должно быть добавлено поле:
- `status` — статус эксперимента.

Возможные значения:
- `CREATED`;
- `RUNNING`;
- `COMPLETED`;
- `FAILED`.

#### 4.4. Поле validation split
Так как пользовательский интерфейс предусматривает размер валидационной выборки, в таблицу `training_config` должно быть добавлено поле:
- `validation_split` — доля валидационной выборки.

#### 4.5. Связь модели с экспериментом
Для однозначной трассировки рекомендуется добавить:
- `experiment_id` в таблицу `model`.

#### 4.6. Таблица `classification_request`
Назначение: хранение истории запросов на классификацию.

Поля:
- `request_id` — первичный ключ;
- `user_id` — внешний ключ на `user`;
- `model_id` — внешний ключ на `model`;
- `image_id` — внешний ключ на таблицу изображений;
- `status` — статус запроса;
- `created_at` — дата и время запроса;
- `completed_at` — дата и время завершения;
- `predicted_class` — итоговый класс;
- `confidence` — итоговая уверенность;
- `raw_response` — сериализованный ответ Python-сервиса.

#### 4.7. Таблица `uploaded_image`
Назначение: хранение метаданных загруженных изображений.

Поля:
- `image_id` — первичный ключ;
- `user_id` — внешний ключ на `user`;
- `original_filename` — исходное имя файла;
- `stored_filename` — имя файла в хранилище;
- `content_type` — MIME-тип;
- `size_bytes` — размер файла;
- `storage_path` — путь хранения;
- `uploaded_at` — дата загрузки.

#### 4.8. Таблица `classification_probability`
Назначение: нормализованное хранение вероятностей по классам, если не используется JSON-поле.

Поля:
- `probability_id` — первичный ключ;
- `request_id` — внешний ключ на `classification_request`;
- `class_name` — имя класса;
- `probability_value` — вероятность.

### 5. Требования к ключам и ограничениям
1. Для всех таблиц должны быть определены первичные ключи.
2. Для внешних ссылок должны быть определены внешние ключи.
3. Для уникальных полей `username` и `email` должны быть заданы ограничения уникальности.
4. Для обязательных полей должны быть заданы ограничения `NOT NULL`.
5. Для ролей должен быть обеспечен контроль допустимых значений.

### 6. Требования к хранению паролей
1. Пароли не должны храниться в открытом виде.
2. В базе данных должен храниться только хэш пароля.
3. Для хэширования должен использоваться современный алгоритм, поддерживаемый Spring Security, например BCrypt.

### 7. Требования к хранению файлов
1. Изображения не рекомендуется хранить как BLOB в PostgreSQL в базовой версии курсовой.
2. Файлы изображений должны сохраняться в файловой системе backend или в выделенном локальном каталоге.
3. В базе данных должны храниться только метаданные файла и путь хранения.
4. Файлы `model.pth` не хранятся в базе данных Java backend.

### 8. Требования к согласованности данных
1. Эксперимент не может ссылаться на несуществующий датасет.
2. Эксперимент не может ссылаться на несуществующую конфигурацию.
3. Модель не может использоваться для классификации, если она не связана с корректным экспериментом и не имеет статуса доступности.
4. История классификации не может существовать без пользователя, модели и изображения.

### 9. Требования к миграциям базы данных
Для управления схемой базы данных должна использоваться миграционная стратегия.

Допускаются варианты:
- Flyway;
- Liquibase;
- генерация схемы Hibernate только для демонстрационного режима.

Для курсовой предпочтительно использовать Flyway либо Liquibase.

### 10. Требования к начальному наполнению базы
В демонстрационной версии должны быть предусмотрены:
- начальные роли `ROLE_USER` и `ROLE_ADMIN`;
- не менее одного администратора;
- не менее одного тестового датасета;
- при необходимости — несколько демонстрационных моделей и метрик.

# Техническое задание
## Часть 5. REST API, валидация и обработка ошибок

### 1. Общие требования к API
1. Backend должен предоставлять REST API.
2. Все данные должны передаваться в формате JSON, кроме загрузки файлов, где используется `multipart/form-data`.
3. Все защищенные эндпоинты должны требовать JWT-аутентификацию.
4. API должно быть задокументировано через Swagger / OpenAPI.

### 2. Группы REST-эндпоинтов
#### 2.1. Аутентификация
- `POST /api/auth/register` — регистрация пользователя;
- `POST /api/auth/login` — вход пользователя и получение JWT;
- `GET /api/auth/me` — получение данных текущего пользователя.

#### 2.2. Пользователи
- `GET /api/users` — список пользователей, только для администратора;
- `GET /api/users/{id}` — карточка пользователя;
- `PUT /api/users/{id}` — изменение пользователя;
- `DELETE /api/users/{id}` — удаление пользователя.

#### 2.3. Датасеты
- `GET /api/datasets` — список датасетов;
- `GET /api/datasets/{id}` — детали датасета.

#### 2.4. Конфигурации обучения
- `POST /api/configs` — создание конфигурации;
- `GET /api/configs/{id}` — получение конфигурации;
- `PUT /api/configs/{id}` — изменение конфигурации, если допускается сценарием;
- `DELETE /api/configs/{id}` — удаление конфигурации, если она не используется.

#### 2.5. Эксперименты
- `POST /api/experiments` — создание эксперимента;
- `GET /api/experiments` — список экспериментов;
- `GET /api/experiments/{id}` — детали эксперимента;
- `GET /api/experiments/{id}/report` — отчет эксперимента;
- `PATCH /api/experiments/{id}/status` — изменение статуса, если используется внутренний или mock-механизм.

#### 2.6. Модели
- `GET /api/models` — список моделей;
- `GET /api/models/{id}` — карточка модели;
- `GET /api/models/{id}/metrics` — метрики модели.

#### 2.7. Классификация
- `POST /api/classifications` — запуск классификации по изображению;
- `GET /api/classifications` — история классификаций текущего пользователя либо всех, если администратор;
- `GET /api/classifications/{id}` — детали результата классификации.

#### 2.8. Файлы
- `POST /api/files/images` — загрузка изображения;
- `GET /api/files/images/{id}` — получение изображения или метаданных изображения.

### 3. Требования к структуре запросов и ответов
#### 3.1. Формат успешного ответа
Ответ API должен содержать:
- тело данных;
- HTTP-статус;
- при необходимости — служебные поля пагинации.

#### 3.2. Формат ответа с ошибкой
Для единообразной обработки ошибок должен использоваться единый формат, содержащий как минимум:
- `timestamp`;
- `status`;
- `error`;
- `message`;
- `path`;
- `validationErrors`, если ошибка вызвана нарушением ограничений входных данных.

### 4. Требования к HTTP-статусам
Необходимо использовать корректные коды ответа:
- `200 OK` — успешный запрос;
- `201 Created` — успешное создание сущности;
- `204 No Content` — успешное удаление без тела ответа;
- `400 Bad Request` — ошибка валидации или некорректный запрос;
- `401 Unauthorized` — отсутствие или некорректность JWT;
- `403 Forbidden` — недостаточность прав;
- `404 Not Found` — объект не найден;
- `409 Conflict` — конфликт данных, например дублирование username или email;
- `500 Internal Server Error` — непредвиденная ошибка сервера;
- `502 Bad Gateway` или `503 Service Unavailable` — ошибка внешнего Python-сервиса, если она проксируется наружу.

### 5. Требования к JWT-аутентификации
1. После успешного входа backend должен возвращать JWT-токен.
2. Angular frontend должен передавать токен в заголовке `Authorization: Bearer <token>`.
3. Защищенные эндпоинты должны быть недоступны без валидного токена.
4. Роль пользователя должна учитываться при авторизации запросов.
5. Для учебной реализации допускается stateless-авторизация без хранения сессий.

### 6. Требования к валидации входных данных
#### 6.1. Пользователь
- `username` — обязателен, не пустой, уникальный, ограниченный по длине;
- `email` — обязателен, валидный email, уникальный;
- `password` — обязателен, не пустой, ограниченный минимальной длиной.

#### 6.2. Конфигурация обучения
- `epochsNum` > 0;
- `batchSize` > 0;
- `learningRate` > 0;
- `optimizer` — обязателен;
- `lossFunction` — обязателен;
- `validationSplit` — значение в допустимом диапазоне.

#### 6.3. Эксперимент
- `datasetId` — обязателен и должен ссылаться на существующий датасет;
- `config` либо `configId` — обязателен;
- пользователь должен быть аутентифицирован.

#### 6.4. Классификация
- `modelId` — обязателен;
- файл изображения — обязателен;
- формат файла должен быть допустимым;
- размер файла не должен превышать лимит.

### 7. Требования к пагинации
Для списков пользователей, экспериментов, моделей и классификаций должна быть предусмотрена пагинация.

Рекомендуемые параметры:
- `page`;
- `size`;
- `sort`.

### 8. Требования к сортировке и фильтрации API
API должно поддерживать:
- фильтрацию экспериментов по статусу;
- фильтрацию экспериментов по `userId`;
- фильтрацию моделей по `datasetId`;
- поиск пользователей по `username`;
- фильтрацию истории классификаций по дате.

### 9. Требования к документации API
Swagger / OpenAPI должен содержать:
- описание всех эндпоинтов;
- описание схем запросов и ответов;
- описание JWT-аутентификации;
- примеры запросов;
- коды ответов;
- описание ошибок.

### 10. Требования к глобальной обработке ошибок
В backend должен быть реализован глобальный обработчик исключений.

Он должен обрабатывать как минимум:
- ошибки валидации;
- ошибки отсутствия сущности;
- ошибки нарушения уникальности;
- ошибки доступа;
- ошибки интеграции с внешним сервисом;
- непредвиденные ошибки приложения.

### 11. Требования к загрузке файлов
1. Загрузка должна выполняться через `multipart/form-data`.
2. Backend должен проверять MIME-тип и расширение файла.
3. Backend должен генерировать безопасное внутреннее имя файла.
4. Backend не должен использовать исходный путь, переданный клиентом.
5. После загрузки должно создаваться описание файла в базе данных.

### 12. Требования к безопасности API
1. Публичными являются только эндпоинты регистрации и входа.
2. Эндпоинты управления пользователями доступны только `ROLE_ADMIN`.
3. Эндпоинты создания эксперимента и классификации доступны аутентифицированным пользователям.
4. Пользователь не должен иметь доступ к чужим данным, если это отдельно не разрешено администратору.

# Техническое задание
## Часть 6. Нефункциональные требования, развертывание и критерии приемки

### 1. Нефункциональные требования к производительности
1. Система должна обеспечивать приемлемое время ответа интерфейса при типовых учебных нагрузках.
2. Операции чтения списков пользователей, экспериментов и моделей должны выполняться без заметных задержек при объеме данных, характерном для курсовой работы.
3. Длительные операции обучения не должны блокировать работу пользовательского интерфейса.
4. Операции классификации должны иметь статус выполнения и возвращать результат после ответа Python-сервиса.

### 2. Требования к надежности
1. Ошибка одного пользовательского запроса не должна приводить к остановке приложения.
2. При недоступности Python-сервиса backend должен возвращать контролируемую ошибку.
3. Поврежденный или недопустимый файл не должен сохраняться как валидный результат загрузки.
4. Данные пользователей и экспериментов должны сохраняться в PostgreSQL.

### 3. Требования к безопасности
1. Пароли должны храниться только в хэшированном виде.
2. API должно использовать JWT-аутентификацию.
3. Должно быть реализовано разграничение доступа по ролям.
4. Необходимо исключить прямой доступ неавторизованного пользователя к защищенным ресурсам.
5. Необходимо обеспечить базовую защиту от типовых ошибок авторизации и некорректного доступа к данным.
6. Должна быть реализована серверная валидация входных данных.

### 4. Требования к сопровождаемости
1. Код backend должен быть разделен на слои: controller, service, repository, entity, dto, security, integration.
2. Код frontend должен быть разделен на модули, компоненты и сервисы.
3. Конфигурация внешних сервисов должна выноситься в настройки приложения.
4. Структура проекта должна быть пригодна для расширения mock-адаптеров на реальные интеграции.

### 5. Требования к развертыванию
Для локального запуска должна быть подготовлена среда Docker Compose.

Рекомендуемый состав контейнеров:
- `postgres` — база данных PostgreSQL;
- `backend` — Spring Boot приложение;
- `frontend` — Angular приложение;
- опционально `mock-python-service` — mock-реализация внешнего сервиса.

### 6. Требования к конфигурации приложения
В конфигурации должны быть предусмотрены параметры:
- строка подключения к PostgreSQL;
- логин и пароль к PostgreSQL;
- настройки JWT;
- настройки директорий хранения файлов;
- URL сервисов обучения и классификации;
- флаг mock-режима;
- настройки логирования.

### 7. Требования к тестированию
Минимально должны быть предусмотрены:
- модульные тесты для сервисного слоя backend;
- интеграционные тесты для REST API ключевых сценариев;
- тесты валидации и безопасности для основных защищенных эндпоинтов.

### 8. Требования к пользовательскому интерфейсу
Интерфейс должен обеспечивать:
- понятную навигацию между разделами;
- разделение экранов входа, регистрации, экспериментов и классификации;
- вывод ошибок валидации пользователю;
- отображение статусов экспериментов;
- отображение метрик моделей;
- отображение результата классификации в наглядном виде.

### 9. Требования к совместимости
1. Backend должен запускаться на Java 17.
2. Backend должен разрабатываться на Spring Boot 4.x.
3. Клиент должен работать как SPA-приложение на Angular.
4. База данных должна быть PostgreSQL.

### 10. Состав поставки в рамках курсовой работы
В состав результата должны входить:
- исходный код backend;
- исходный код frontend;
- скрипты миграций базы данных;
- docker-compose файл;
- документация по запуску;
- Swagger / OpenAPI документация;
- тестовые данные либо seed-скрипты.

### 11. Критерии приемки. Общие
Работа считается выполненной, если:
1. приложение запускается локально;
2. backend подключается к PostgreSQL;
3. frontend взаимодействует с backend через REST API;
4. реализована регистрация и вход пользователя;
5. реализована JWT-авторизация;
6. реализован доступ по ролям;
7. реализовано создание эксперимента;
8. реализован просмотр списка и карточки эксперимента;
9. реализован просмотр моделей и метрик;
10. реализована загрузка изображения и классификация;
11. реализовано администрирование пользователей;
12. реализованы обработка ошибок и серверная валидация;
13. API документировано в Swagger.

### 12. Критерии приемки. По функциональности
#### 12.1. Регистрация и вход
Критерий приемки:
- новый пользователь может зарегистрироваться;
- пользователь может войти по логину и паролю;
- после входа получает JWT;
- без JWT защищенные разделы недоступны.

#### 12.2. Управление пользователями
Критерий приемки:
- администратор видит список пользователей;
- администратор может изменить и удалить пользователя;
- обычный пользователь не имеет доступа к административному разделу.

#### 12.3. Эксперименты
Критерий приемки:
- пользователь может выбрать датасет и параметры обучения;
- пользователь может создать эксперимент;
- эксперимент отображается в общем списке;
- карточка эксперимента показывает статус, конфигурацию и метрики.

#### 12.4. Классификация
Критерий приемки:
- пользователь может выбрать модель;
- пользователь может загрузить изображение;
- пользователь может получить прогноз;
- результат содержит класс и вероятности.

#### 12.5. Валидация и ошибки
Критерий приемки:
- при пустых или некорректных данных система возвращает понятные ошибки;
- при отсутствии прав система возвращает `403 Forbidden`;
- при ошибке токена система возвращает `401 Unauthorized`.

### 13. Ограничения учебной реализации
1. Производительность оценивается на учебных данных.
2. Поддержка распределенного хранения файлов не требуется.
3. Реальная промышленная оркестрация контейнеров не требуется.
4. Реальные Python-компоненты могут быть заменены на mock-реализацию без нарушения общей архитектуры.
